# Documentation

This folder is the index for the working documentation that sits next to the code. It is meant to stay short and readable. The goal is not to duplicate every GitHub issue in prose, but to keep the core architectural decisions and repository conventions in one place so the codebase can be understood without reconstructing context from project boards.

Start with [architecture.md](architecture.md) if you want the product and service model, then [repository-layout.md](repository-layout.md) if you want the codebase shape. [deployment-baseline.md](deployment-baseline.md) describes how the repo is prepared for Pages, Railway, and worker-image builds, and now also records the baseline rollback path for each deployable surface without overcommitting early implementation details. [docker-image-baseline.md](docker-image-baseline.md) defines the offline `firstproof/Problem9` devbox and execution-image policy, runtime-tool boundary, and never-in-image secret rules. [lean-mathlib-version-baseline.md](lean-mathlib-version-baseline.md) defines the canonical Lean lanes, Mathlib pinning rules, mixed-version handling, and version-change policy for the offline `firstproof/Problem9` slice. [problem9-benchmark-package-baseline.md](problem9-benchmark-package-baseline.md) defines the immutable package root, required files, and checksum contract for the offline `firstproof/Problem9` benchmark slice. [problem-9-verifier-golden-baseline.md](problem-9-verifier-golden-baseline.md) defines the canonical positive goldens, required negative fixtures, theorem-equality recording, and local plus CI verifier gates for the Problem 9 slice. [problem9-run-bundle-baseline.md](problem9-run-bundle-baseline.md) defines the canonical offline run bundle, verifier verdict subset, ingest-critical files, and digest boundary for the Problem 9 slice. [problem9-agent-loop-baseline.md](problem9-agent-loop-baseline.md) defines the supported Problem 9 run modes, bounded repair-loop state machine, whole-attempt retry boundary, and downstream implications for budgets, verifier policy, and worker events. [problem9-lean-run-strategy-baseline.md](problem9-lean-run-strategy-baseline.md) defines the authoritative Lean validation pipeline, theorem-drift checks, axiom allowlist policy, and proof-acceptance rules for the Problem 9 slice. [problem9-reproducibility-baseline.md](problem9-reproducibility-baseline.md) defines the mandatory reproducibility tuple, model snapshot tracking rules, environment digest requirements, and run-configuration identity for the Problem 9 slice. [problem9-failure-registration-baseline.md](problem9-failure-registration-baseline.md) defines the canonical Problem 9 failure classes, retryability split, evidence requirements, and user-visible failure boundary for offline runs. [internal-contracts-api-baseline.md](internal-contracts-api-baseline.md) defines the MVP browser/API/worker contract split, synchronous versus asynchronous flow boundary, and canonical run/job/attempt state transitions. [provider-framework-api-baseline.md](provider-framework-api-baseline.md) defines the normalized provider adapter request/response boundary, auth-mode split, required provider tiers, and escape-hatch rules for OpenAI, Aristotle, Anthropic, Google, Axle, and future backends. [model-config-registry-baseline.md](model-config-registry-baseline.md) defines the repository-owned `modelConfigId` contract, snapshot pinning rules, reasoning-mode defaults, and central-versus-run-level limit boundary. [email-strategy-baseline.md](email-strategy-baseline.md) defines the MVP decision to keep app email out of scope for now, split support mail from transactional mail, and use Resend as the default later outbound provider. [better-auth-migration-baseline.md](better-auth-migration-baseline.md) defines the post-MVP boundary between Cloudflare Access, Better Auth, and ParetoProof-owned authorization data. [capacity-baseline.md](capacity-baseline.md) defines the MVP load assumptions, first bottlenecks, and scaling rules for Railway, Neon, Modal, and R2. [operations-baseline.md](operations-baseline.md) records the service ownership, hostname, and secret model that the current roadmap relies on. [github-environment-secrets-baseline.md](github-environment-secrets-baseline.md) defines the staging and production GitHub Actions secret boundary and bootstrap flow. [frontend-design-system-baseline.md](frontend-design-system-baseline.md) defines the MVP component stack, styling-token rules, responsive constraints, and routing composition style for `apps/web`. [results-drilldown-ux-baseline.md](results-drilldown-ux-baseline.md) defines the MVP results pivots, failure/rerun detail visibility, comparison baseline, and shareable drilldown state in the portal UX. [release-version-mapping-baseline.md](release-version-mapping-baseline.md) defines how Git SHAs map to deploy artifacts and environments for traceable rollback and restore. [r2-retention-restore-baseline.md](r2-retention-restore-baseline.md) defines the MVP R2 retention windows, object mutability rules, and owner-led recovery process for artifacts and exports. [restore-drill-checklist-baseline.md](restore-drill-checklist-baseline.md) defines the quarterly MVP recovery drill checklist for Neon, R2, and deploy artifact traceability. [observability-baseline.md](observability-baseline.md) defines the MVP logs, metrics, traces, dashboard, alert, and cost/rate-limit monitoring baseline. [run-control-governance-baseline.md](run-control-governance-baseline.md) defines MVP retry, cancel, concurrency, and budget enforcement rules for control-plane run execution. [prompt-run-protocol-baseline.md](prompt-run-protocol-baseline.md) defines the MVP prompt layers, context package, tool-permission profiles, and Lean-MCP usage policy for worker attempts. [data-baseline.md](data-baseline.md) captures the first database entity and branch strategy decisions. [testing-baseline.md](testing-baseline.md) defines the MVP test stack choice and required testing methodology. [project-management.md](project-management.md) explains how scoping issues and execution issues are meant to move through GitHub. The accepted implementation stack decision is recorded in [adrs/0001-mvp-implementation-stack.md](adrs/0001-mvp-implementation-stack.md) and the hosting choice in [adrs/0002-mvp-hosting-decision.md](adrs/0002-mvp-hosting-decision.md).
