# Operations Baseline

The repository needs one operational baseline that tells contributors which platform owns each surface, which hostnames are intended to exist, and where secrets are supposed to live. The goal is not to mirror every dashboard setting locally. The goal is to make the ownership model clear enough that later setup issues can be completed without reopening the same questions.

In local development, the web app, API, and worker run from the Bun workspace on a developer machine. The default local database path is a local Postgres instance, with Neon branches reserved for parity testing and shared debugging rather than everyday local development. In staging, the intended shape is a Cloudflare-hosted web surface, a Railway API environment, a Neon staging branch, Modal worker execution, and R2 storage with staging-separated object paths. In production, Cloudflare owns the public edge, Railway owns the control plane, Neon owns structured data, Modal owns worker execution, and R2 owns large artifacts.

The hostname plan follows that split. `paretoproof.com` is the public root. `www.paretoproof.com` is treated as a redirect or alias to the public site. `portal.paretoproof.com` is the authenticated portal surface. `api.paretoproof.com` is the control-plane API. Local development, not a separate staging hostname, is the pre-production path for ongoing frontend work. `math.paretoproof.com` remains intentionally out of current MVP scope. No public worker hostname is part of the plan.

The current account ownership model is simple. The repository owner controls the GitHub, Cloudflare, Railway, Neon, and Modal accounts for the MVP. That means platform-level billing, domain ownership, bootstrap credentials, and service-account creation are all centralized for now. The application itself will later expose narrower collaborator roles, but the platform control plane is still owner-managed.

The secret model follows the same separation. Bootstrap and platform secrets stay outside the repository and are entered into platform-specific secret stores when needed. GitHub holds repository and workflow secrets. Cloudflare holds edge and Access secrets. Railway holds API runtime secrets such as `DATABASE_URL`, session material, and backend signing secrets. Modal holds worker runtime secrets. Neon issues the database credentials but is not the place where application secrets are inventoried. The repository can document secret names and ownership, but it must never hold live secret values.

The current human-controlled bootstrap vault root is `U:\Personal\Secrets\ParetoProof`. That location is the local source of truth for bootstrap secret references, recovery-material placeholders, and runtime-secret location notes. Live values remain outside the repository and are only mirrored into platform secret stores when a service actually needs them.

For the MVP, the current secret inventory is small and stable enough to name explicitly. Bootstrap control-plane access uses `GH_TOKEN`, `CLOUDFLARE_GLOBAL_API_KEY`, `CLOUDFLARE_EMAIL`, `CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_ZONE_ID`, `RAILWAY_API_TOKEN`, `NEON_API_KEY`, `NEON_ORG_ID`, `MODAL_TOKEN_ID`, and `MODAL_TOKEN_SECRET`. Runtime secrets that are expected later include `DATABASE_URL`, `APP_SESSION_SECRET`, `WORKER_BOOTSTRAP_TOKEN`, `R2_ACCESS_KEY_ID`, and `R2_SECRET_ACCESS_KEY`. Model-provider secrets remain later-scope runtime inputs and are not part of the current foundation work.

R2 should follow one naming scheme across all environments. Private run data lives in `paretoproof-dev-artifacts`, `paretoproof-staging-artifacts`, and `paretoproof-production-artifacts`. Generated bundles that may later need different access policy live in `paretoproof-dev-exports`, `paretoproof-staging-exports`, and `paretoproof-production-exports`. That keeps lifecycle and credentials scoped per environment without creating a separate bucket for every artifact class.

Prefixes stay stable inside those buckets. The artifacts buckets use `runs/<run_id>/artifacts/`, `runs/<run_id>/logs/`, and `runs/<run_id>/traces/` for execution outputs, with any later benchmark-source payloads placed under `benchmarks/<benchmark_version_id>/source/`. The exports buckets use `runs/<run_id>/bundles/` for downloadable run packages and `benchmarks/<benchmark_version_id>/reports/` for published result bundles. Buckets should be created with EU jurisdiction so storage residency matches the current hosting bias, because Cloudflare treats jurisdiction choice as immutable once a bucket exists.
